01 — Problem Description

What Problem Are We Solving?

The Challenge

Many hospital patient registration systems struggle to simultaneously track recently registered patients and manage both standard and emergency queues effectively. Without a structured approach, workflows break down — critical patients may wait behind less-urgent arrivals, registration logs lack accountability, and staff have no clear way to audit recent activity.

This system addresses that gap by implementing a multi-level queuing mechanism with triage-based categorization using core data structures covered in CC 103.

Pain PointOur Solution
🔴 Disorganized patient workflowsSeverity-routed dual-queue system (Priority Queue + FIFO Queue)
🔴 Critical patients not prioritizedEmergency PQ always drained before standard queue
🔴 No registration history / accountabilityLIFO Stack maintains a timestamped activity log
🔴 Static display logic (no recursion)Recursive traversal for Priority Queue display
02 — Data Structures Used

How We Organize Patients

🧱

Stack

What it is: A Last-In, First-Out (LIFO) linear structure backed by a std::vector. The most recently pushed element is always the one popped first.

Used for: Recent Registration Log — lets staff instantly review or correct the most recently admitted patient, maintaining an ordered audit trail.

📋

FIFO Queue

What it is: A First-In, First-Out linear structure backed by a std::deque. Patients are served strictly in arrival order.

Used for: Standard Waiting List (Severity 3–5) — ensures fairness; non-critical patients are called in the order they registered.

⚡

Priority Queue

What it is: A sorted singly-linked list where the node with the lowest severity number (highest urgency) is always at the head. Ties preserve arrival order.

Used for: Emergency Triage (Severity 1–2) — critical patients are always seen first, regardless of when they arrived.

Severity Scale

1 — Critical 2 — Serious 3 — Moderate 4 — Minor 5 — Routine

Severity 1–2 routes to the Emergency Priority Queue. Severity 3–5 routes to the Standard FIFO Queue. All patients are always pushed onto the Activity Stack.

03 — Algorithm Explanation

Step-by-Step Logic

01

Patient Registration

The user enters the patient's full name, chief complaint, and severity level (1–5). A unique auto-incremented ID is assigned starting from 1001.

02

Stack Logging (Always)

Every admitted patient is immediately pushed onto the activity vector-backed stack. This is unconditional — it provides an auditable LIFO history regardless of severity.

03

Severity-Based Routing

If severity ≤ 2 → emergencyPQ_.enqueue(patient). If severity ≥ 3 → standardQueue_.push_back(patient). This bifurcation ensures critical cases never compete with routine ones.

04

Priority Queue Insertion (Linked List Sort)

During enqueue(), the linked list is walked until a node with a strictly higher severity number is found. The new node is spliced in before it — producing a stable sorted list. The head always points to the most critical patient.

// Walk until next node has lower priority
while (current->next != nullptr &&
       current->next->data.severity <= patient.severity) {
    current = current->next;
}
// Splice new node between current and current->next
newNode->next  = current->next;
current->next  = newNode;
05

Calling the Next Patient (Drain Order)

When a patient is called, callNextPatient() first checks emergencyPQ_. Only if it is empty does it pull from standardQueue_. This guarantees emergencies always take precedence.

06

Recursive Waitlist Display

The Priority Queue's display() delegates to the recursive helper displayRecursive(Node* current, int pos). It walks the linked list node-by-node: base case — stop when current == nullptr; recursive case — print the current patient row, then call self on current->next.

static void displayRecursive(Node* current, int pos) {
    if (current == nullptr) return; // BASE CASE
    // ... print row for current->data ...
    displayRecursive(current->next, pos + 1); // RECURSIVE CASE
}
04 — Iterative vs Recursive Comparison

Which Approach is Better?

Iterative Original
SpeedSlightly faster — no function call overhead or stack frame allocation per node
MemoryO(1) extra space — uses a single traversal pointer on the heap
ReadabilityFamiliar while loop — easy to follow for most programmers
RiskLow — no risk of call-stack overflow even for very long queues
Recursive Current
SpeedMarginally slower — each call pushes a new frame onto the call stack
MemoryO(n) call-stack depth — one frame per node in the linked list
ReadabilityMore expressive for tree/list traversal — mirrors the structure of the data
RiskLow for typical hospital queue sizes; stack overflow only for extremely large n

Verdict

Iterative is faster and safer in production — zero call-stack overhead and no overflow risk regardless of queue length. However, recursive is easier to understand conceptually for linked list traversal because the code directly mirrors how the list is structured: "process this node, then do the same for the rest." For the typical scale of a hospital triage queue, the performance difference is negligible, making recursion a valid and instructive choice for this academic project.

05 — Design Decisions

Choices & Trade-Offs

Linked List for Priority Queue (not std::priority_queue)

Why: Implementing the PQ from scratch as a singly-linked list gave us direct control over insertion logic and made stable tie-breaking (arrival order) trivial to implement. Using the STL's heap-based priority_queue would have abstracted away the data structure concepts the course requires us to demonstrate.

Trade-off: O(n) insertion vs O(log n) for a heap, but for small-to-medium hospital queues this is acceptable. We gain transparency and educational value.

Dual-Queue Architecture (Emergency PQ + Standard FIFO)

Why: Separating emergency (severity 1–2) and standard (severity 3–5) patients into two distinct structures prevents critical patients from being mixed into a single sorted list with routine ones. The drain-emergency-first rule in callNextPatient() is then trivial to enforce.

Trade-off: Slightly more complex routing logic at admission, but much cleaner dequeue logic and clearer separation of concerns.

std::vector as Stack, std::deque as Queue

Why: Rather than building a stack and FIFO queue from scratch (which would duplicate the linked-list exercise already done for PQ), we leveraged the STL for the logging stack and standard waiting list. push_back/pop_back on a vector is semantically identical to a stack; push_back/pop_front on a deque is a textbook FIFO queue.

Trade-off: Less hand-crafted code, but faster iteration and lower bug surface — appropriate given the custom PQ already demonstrates manual data structure implementation.

Recursive Display as a Static Helper

Why: The displayRecursive function is declared static (file-scope) rather than as a class member. This keeps the Node internals encapsulated — external code never needs to call the helper directly. The public display() method owns the table header/footer and delegates only the row iteration to the recursive helper.

Trade-off: A slight separation between the header logic (iterative) and row logic (recursive), but this results in a cleaner public API.
